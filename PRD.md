# Product Requirements Document: SPK Creator

**Version:** 1.0  
**Last Updated:** January 21, 2026  
**Status:** Draft  
**Type:** Proof of Concept (PoC)

---

## 1. Overview

**SPK Creator** is an internal tool designed to digitize vendor work orders (SPK) and enable transparent payment tracking. This is a proof of concept to validate the core workflow and technical approach before building a production-ready system.

### What is SPK?

SPK (Surat Perintah Kerja) is a work order document issued to vendors that outlines project scope, contract value, and payment terms.

---

## 2. Goals & Objectives

### Primary Goals

1. **Digitize SPK creation** ‚Äì Replace manual/spreadsheet-based work orders with a structured digital form
2. **Transparent payment tracking** ‚Äì Enable real-time visibility into payment status across all stakeholders
3. **Automate notifications** ‚Äì Reduce manual communication via automated Slack/Email alerts
4. **Generate official PDFs** ‚Äì Produce standardized SPK documents directly from the application

### Success Criteria (PoC)

- ‚úÖ Internal users can create and publish SPKs in under 5 minutes
- ‚úÖ Vendors can view their SPKs and payment status without manual updates
- ‚úÖ PDF generation works reliably with correct data
- ‚úÖ n8n automation triggers successfully on key events

---

## 3. User Personas

### Internal User (SPK Creator)

- **Role:** Project Manager / Finance Team
- **Needs:** Create SPKs quickly, track payment status, send updates to vendors
- **Access Level:** Full CRUD on SPKs and payments

### Vendor (Read-Only User)

- **Role:** External contractor/supplier
- **Needs:** View their SPKs, check payment status, download PDF copies
- **Access Level:** Read-only access to their own SPKs

## 4. Core Features

### 4.1 SPK Creation Form

**Description:** A multi-step or single-page form to create new SPK records.

**Required Fields:**

- Vendor name
- Vendor contact (email, phone - optional but recommended)
- Project name/description
- SPK number (auto-generated by backend, editable by user for manual override)
- Contract value (total amount)
- Currency (selectable from multiple options: IDR, USD, SGD, EUR, etc.)
- Start date
- End date (optional)
- Payment terms (dynamic multi-payment):
  - Can add/remove payment rows (1x, 2x, 3x, etc.)
  - Each payment: term name, percentage or nominal amount
  - Flexible structure, no fixed 3-payment limit
- Additional notes/attachments (optional)
- SPK Maker/Creator (auto-populated from logged-in user session)

**Validations:**

- If using percentages: payment terms must sum to 100%
- If using nominal amounts: payments should not exceed contract value (warning, not blocker)
- All required fields must be filled
- Vendor email format validation
- SPK number uniqueness (handled by backend to prevent duplicates)

**User Flow:**

1. User clicks "Create SPK"
2. System auto-generates SPK number (editable if needed)
3. Fills out multi-step form (step-by-step progress indicator):
   - Step 1: Vendor & Project Details
   - Step 2: Contract Value & Currency
   - Step 3: Payment Terms (dynamic add/remove)
   - Step 4: Review & Confirm
4. Saves as draft OR publishes immediately
5. On publish ‚Üí PDF generated + share link created
6. User can optionally send email notification to vendor

---

### 4.2 Payment Terms Structure

**Dynamic Multi-Payment System:**

Each SPK can have flexible payment milestones (1x, 2x, 3x, or more):

- User can add or remove payment rows dynamically
- Each payment term includes:
  - Term name/description (e.g., "Down Payment", "Progress 1", "Final")
  - Payment amount: can be entered as **percentage** OR **nominal amount**
  - Status tracking (Pending, Paid, Overdue)
- No strict validation for percentage-to-nominal conversion
- System calculates totals based on input method

**Example Configurations:**

| Configuration | Terms                                           |
| ------------- | ----------------------------------------------- |
| Simple (1x)   | 100% upon completion                            |
| Standard (3x) | DP 30%, Progress 40%, Final 30%                 |
| Complex (5x)  | DP 20%, Progress 1-4 (20% each)                 |
| Mixed         | DP 30% (percentage), Progress IDR 50M (nominal) |

**Payment Statuses:**

- `Pending` ‚Äì Not yet paid
- `Paid` ‚Äì Payment completed
- `Overdue` ‚Äì Past due date (optional for PoC)

**User Actions:**

- Mark payment as paid (manual update)
- Add payment date
- Add payment reference/proof (optional text field)

---

### 4.3 Invoice/SPK Preview & Sharing

**Description:** Generate and share official SPK PDF documents with preview and shareable links.

**Features:**

- **PDF Preview:** In-app preview before downloading
- **Shareable Link:** Generate unique link to view/download SPK
- **Email Notification:** Optional (not automatic) - user can choose to send via email
- **Download:** Direct PDF download for internal users and vendors

**Technical Approach:**

- Use `@react-pdf/renderer` for server-side PDF generation
- Generate PDFs on-demand (no storage required initially)
- Public API endpoint for shareable links: `/api/pdf/[id]?token=xxx`

**PDF Content:**

- Company header/logo (optional)
- SPK number and date
- SPK Maker information (auto from logged-in user)
- Vendor details
- Project details
- Contract value and currency
- Dynamic payment breakdown (supports multiple payment terms)
- Terms and conditions (static text)
- Signature placeholders (visual only, not functional)

**User Flow:**

1. User publishes SPK ‚Üí PDF preview available
2. User can copy shareable link or download PDF
3. Optionally: User sends email with link/attachment to vendor
4. Vendor/Internal user accesses via link ‚Üí can preview and download

---

### 4.4 Vendor Dashboard (Read-Only)

**Description:** A dedicated view for vendors to see their SPKs and payment status.

**Access Method:**

- Simple authentication via unique link (e.g., `/vendor/[vendorId]?token=xyz`)
- OR basic email/password login (optional for PoC)

**Dashboard Contents:**

- List of all SPKs for this vendor
- SPK details (project, value, dates)
- Payment status breakdown (DP, Progress, Final)
- Download PDF button
- Last updated timestamp

**No Edit Access:** Vendors cannot create or modify SPKs.

### 4.5 Payment Status Tracking

**Description:** Internal dashboard to view and update payment statuses across all SPKs.

**Features:**

- Table view of all SPKs with filters:
  - By vendor
  - By status (draft, published)
  - By payment status (paid, pending, overdue)
- Quick actions:
  - Mark payment as paid
  - View SPK details
  - Download PDF
- Status indicators (color-coded badges)

**Update Flow:**

1. Internal user marks payment as "Paid"
2. Payment date and reference recorded
3. n8n webhook triggered ‚Üí notification sent

```sql
-- Enable UUID extension
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- SPK Table
CREATE TABLE spk (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  spk_number VARCHAR(50) UNIQUE NOT NULL,
  vendor_name VARCHAR(255) NOT NULL,
  vendor_email VARCHAR(255),
  vendor_phone VARCHAR(50),
  project_name VARCHAR(255) NOT NULL,
  project_description TEXT,
  contract_value NUMERIC(15, 2) NOT NULL,
  currency VARCHAR(10) DEFAULT 'IDR', -- Supports: IDR, USD, SGD, EUR, MYR, etc.
  start_date DATE NOT NULL,
  end_date DATE,

  status VARCHAR(20) CHECK(status IN ('draft', 'published')) DEFAULT 'draft',
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  created_by VARCHAR(255) NOT NULL, -- Auto from session: user name
  created_by_email VARCHAR(255) NOT NULL, -- Auto from session: user email
  notes TEXT,
  pdf_url TEXT
);

-- Payment Table (Dynamic Multi-Payment Support)
CREATE TABLE payment (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  spk_id UUID NOT NULL REFERENCES spk(id) ON DELETE CASCADE,
  term_name VARCHAR(100) NOT NULL, -- e.g., "Down Payment", "Progress 1", "Final"
  term_order INTEGER NOT NULL, -- Sequence: 1, 2, 3, ...
  amount NUMERIC(15, 2) NOT NULL,
  percentage NUMERIC(5, 2), -- Optional: if user entered as percentage
  input_type VARCHAR(20) CHECK(input_type IN ('percentage', 'nominal')) NOT NULL,
  status VARCHAR(20) CHECK(status IN ('pending', 'paid', 'overdue')) DEFAULT 'pending',
  due_date DATE,
  paid_date DATE,
  payment_reference VARCHAR(255),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_by VARCHAR(255) NOT NULL,
  UNIQUE(spk_id, term_order) -- Prevent duplicate orders for same SPK
);

-- Vendor Table (Optional)
CREATE TABLE vendor (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  name VARCHAR(255) NOT NULL,
  email VARCHAR(255),
  phone VARCHAR(50),
  access_token TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Indexes for better query performance
CREATE INDEX idx_spk_vendor_name ON spk(vendor_name);
CREATE INDEX idx_spk_status ON spk(status);
CREATE INDEX idx_spk_created_at ON spk(created_at);
CREATE INDEX idx_payment_spk_id ON payment(spk_id);
CREATE INDEX idx_payment_status ON payment(status);
CREATE INDEX idx_vendor_email ON vendor(email);

-- Enable Row Level Security (RLS)
ALTER TABLE spk ENABLE ROW LEVEL SECURITY;
ALTER TABLE payment ENABLE ROW LEVEL SECURITY;
ALTER TABLE vendor ENABLE ROW LEVEL SECURITY;

-- RLS Policies (Basic - adjust based on auth strategy)
-- Allow internal users full access (authenticated users)
CREATE POLICY "Allow full access for authenticated users" ON spk
  FOR ALL USING (auth.role() = 'authenticated');

CREATE POLICY "Allow full access for authenticated users" ON payment
  FOR ALL USING (auth.role() = 'authenticated');

-- Allow vendors to view their own SPKs (using vendor_email)
CREATE POLICY "Vendors can view their own SPKs" ON spk
  FOR SELECT USING (vendor_email = auth.jwt() ->> 'email');

CREATE POLICY "Vendors can view payments for their SPKs" ON payment
  FOR SELECT USING (
    spk_id IN (
      SELECT id FROM spk WHERE vendor_email = auth.jwt() ->> 'email'
    )
  );

-- Trigger to update updated_at timestamp
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_spk_updated_at BEFORE UPDATE ON spk
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

CREATE TRIGGER update_payment_updated_at BEFORE UPDATE ON payment
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

### Sample Queries

```sql
-- Get all published SPKs with vendor info
SELECT
  spk_number, vendor_name, project_name, contract_value,
  currency, status, created_at, created_by
FROM spk
WHERE status = 'published'
ORDER BY created_at DESC;

-- Get SPK details with dynamic payment breakdown
SELECT
  s.spk_number, s.vendor_name, s.project_name,
  s.contract_value, s.currency,
  p.term_name, p.term_order, p.amount, p.percentage, p.input_type,
  p.status as payment_status, p.paid_date, p.payment_reference
FROM spk s
LEFT JOIN payment p ON s.id = p.spk_id
WHERE s.spk_number = 'SPK-2026-001'
ORDER BY p.term_order;

-- Get all SPKs for a specific vendor
SELECT
  spk_number, project_name, contract_value, currency, status, created_at
FROM spk
WHERE vendor_name = 'PT Vendor Jaya'
ORDER BY created_at DESC;

-- Get payment summary across all SPKs
SELECT
  status,
  COUNT(*) as payment_count,
  SUM(amount) as total_amount
FROM payment
GROUP BY status;

-- Get pending payments with SPK details
SELECT
  s.spk_number, s.vendor_name, s.project_name,
  p.term_name, p.term_order, p.amount, p.status
FROM payment p
JOIN spk s ON p.spk_id = s.id
WHERE p.status = 'pending' AND s.status = 'published'
ORDER BY p.term_order ASC;

-- Get SPKs with at least one paid payment
SELECT DISTINCT
  s.spk_number, s.vendor_name, s.project_name,
  s.contract_value, s.currency,
  COUNT(CASE WHEN p.status = 'paid' THEN 1 END) as paid_count,
  SUM(CASE WHEN p.status = 'paid' THEN p.amount ELSE 0 END) as paid_total
FROM spk s
JOIN payment p ON s.id = p.spk_id
GROUP BY s.id
HAVING paid_count > 0;

-- Count payment terms per SPK (useful for dynamic payment verification)
SELECT
  s.spk_number,
  s.project_name,
  COUNT(p.id) as total_payment_terms,
  SUM(p.amount) as total_payment_amount,
  s.contract_value,
  (SUM(p.amount) - s.contract_value) as difference
FROM spk s
LEFT JOIN payment p ON s.id = p.spk_id
GROUP BY s.id
ORDER BY s.created_at DESC;
```

#### Example Server Action (with Dynamic Payments)

```typescript
// app/actions/spk.ts
"use server";

import { supabaseAdmin } from "@/lib/supabase/server";
import { revalidatePath } from "next/cache";
import { Resend } from "resend";

const resend = new Resend(process.env.RESEND_API_KEY);

// Helper: Auto-generate SPK number
async function generateSPKNumber(): Promise<string> {
  const year = new Date().getFullYear();
  const { count } = await supabaseAdmin
    .from("spk")
    .select("*", { count: "exact", head: true });

  const nextNumber = (count || 0) + 1;
  return `SPK-${year}-${String(nextNumber).padStart(3, "0")}`;
}

export async function createSPK(data: any, userSession: any) {
  // Auto-generate SPK number if not provided
  const spkNumber = data.spkNumber || (await generateSPKNumber());

  const { data: spk, error } = await supabaseAdmin
    .from("spk")
    .insert({
      spk_number: spkNumber,
      vendor_name: data.vendorName,
      vendor_email: data.vendorEmail,
      project_name: data.projectName,
      contract_value: data.contractValue,
      currency: data.currency,
      start_date: data.startDate,
      status: "draft",
      created_by: userSession.user.name, // From session
      created_by_email: userSession.user.email, // From session
    })
    .select()
    .single();

  if (error) throw error;

  // Create dynamic payment records
  const payments = data.paymentTerms.map((term: any, index: number) => ({
    spk_id: spk.id,
    term_name: term.name,
    term_order: index + 1,
    amount: term.amount,
    percentage: term.percentage,
    input_type: term.inputType, // "percentage" or "nominal"
    status: "pending",
    updated_by: userSession.user.email,
  }));

  await supabaseAdmin.from("payment").insert(payments);

  revalidatePath("/dashboard");
  return spk;
}

export async function publishSPK(spkId: string, sendEmail: boolean = false) {
  // Update SPK status
  const { data: spk } = await supabaseAdmin
    .from("spk")
    .update({ status: "published" })
    .eq("id", spkId)
    .select()
    .single();

  if (!spk) throw new Error("SPK not found");

  // Send Slack notification (always)
  await fetch(process.env.SLACK_WEBHOOK_URL!, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      text: `üéâ New SPK #${spk.spk_number} published for ${spk.vendor_name} - ${spk.project_name}`,
    }),
  });

  // Send email notification (optional)
  if (sendEmail && spk.vendor_email) {
    await resend.emails.send({
      from: "noreply@company.com",
      to: spk.vendor_email,
      subject: `New SPK: ${spk.spk_number}`,
      html: `<p>A new SPK has been created for ${spk.project_name}. <a href="${process.env.APP_URL}/api/pdf/${spk.id}">View PDF</a></p>`,
    });
  }

  revalidatePath("/dashboard");
  return spk;
}
```

## 10. Open Questions & Assumptions

### Assumptions Made

- SPK numbers are auto-generated by backend (e.g., `SPK-2026-001`) but editable by user
- Payment terms are dynamic and flexible (not limited to 3 fixed terms)
- User can enter payment as percentage OR nominal amount
- No automatic currency conversion required (display as entered)
- SPK Maker information auto-populated from logged-in user session
- One internal user role for this phase (no complex permissions)
- Vendors identified by email (no formal vendor master data)
- Email sending is optional and user-triggered (not automatic on SPK creation)

### Questions for Future Clarification

- Should vendors be able to upload payment proof documents?
- Do we need comprehensive audit trail for all changes?
- Should there be automated payment deadline reminders?
- Is there a specific company PDF template/branding required?
- Do we need to support multiple languages for international vendors?

### Key Technical Decisions

- Use Server Actions for mutations (create SPK, update payment)
- Use API routes for PDF generation endpoints
- Auto-populate SPK Maker from user session (not manual input)
- Generate SPK numbers in backend to prevent duplicates
- Support dynamic payment terms (not hardcoded 3-term structure)
- Use Supabase Auth for internal users, token-based access for vendors
- Generate PDFs on-demand (no storage required initially)
- Use Resend for email (direct API, no n8n)
- Use Slack webhooks for notifications (direct API, no n8n)
- Leverage Supabase Row Level Security (RLS) for data access control
